Arms are Kraken's workers ‚Äî and Kraken deliberately **is not** an agent
harness. It doesn't run models, manage context windows, or implement tool
use. It dispatches work to the coding-agent CLIs you already use, and treats
them as interchangeable muscle.

## Built-in and pluggable harnesses

| Arm | Invocation | Notes |
| --- | --- | --- |
| `claude` | `claude -p` (headless) | The default. Native stream parsing, live activity, tier‚Üímodel maps, `--allowedTools` grants. |
| `codex` | `codex exec --json --sandbox workspace-write` | Built in. Tier map defaults to the current lineup (fast/standard/frontier); an `[[arms]]` block named `codex` overrides only the tier models. No per-dispatch tool allow-list flag ‚Äî use `~/.codex/rules` for policy. |
| anything | your command | Any coding CLI wired as a plain subprocess via `[[arms]]` in `kraken.toml`, with per-tier args. |

The **model tier** (`fast` / `standard` / `frontier`) is the contract's
knob, not the harness's: each arm maps tiers to its own models, so a
contract escalating fast ‚Üí frontier means the right thing on every harness.

## Worktree isolation

Every dispatch gets a fresh git worktree on its own branch, cut from trunk.
Arms never share a working directory, never see each other's uncommitted
state, and cannot reach sibling checkouts or other repositories. Cross-arm
context flows through exactly one channel: the
[findings bus](../findings-bus/).

## Harness-owned commits

The arm commits its own work. Kraken doesn't reach into a worktree and
`git add` whatever it finds ‚Äî the harness decides what belongs in the
change, and Kraken verifies from the outside (and treats "reported success
but produced no changes" as a decision-worthy anomaly, not a silent pass).

## Receipts, not claims

Each arm ends by filing a structured report ‚Äî summary, files touched,
decisions, gotchas, interface changes, blockers, confidence. That report is
the arm's **receipt**: the judge reads it as the worker's claim (ranked
below gate evidence), the bus routes its findings to siblings, and campaigns
compose its interface changes into downstream goals. Free-text output is
tolerated as a fallback, never preferred.

## Fail-closed tool access

Arms run non-interactive under a **default-deny** tool policy. Repo-level
`allowed_tools` in `kraken.toml` (Claude Code `--allowedTools` syntax) plus
any per-contract `retry-with-tools` grants are everything an arm may touch
beyond file edits. Anything else is silently denied at the harness level ‚Äî
the worker hits the wall, reports it as a blocker in its receipt, and the
[permission-skip](../judge/#the-permission-skip) turns that into a scoped
grant decision instead of a louder model.

```toml
# kraken.toml ‚Äî grant narrowly, prefer read-only
allowed_tools = ["Bash(gh pr view:*)", "Bash(gh api:*)"]
```

Sandboxing beyond this is deliberately out of scope: Kraken inherits the
harness's sandbox (and, planned, sandbox-runtime) rather than building its
own.
