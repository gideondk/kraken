Kraken never hands an agent a vibe. The planner (a frontier-tier arm reading
your actual repo) decomposes a goal into **task contracts**: typed,
validated units of work with explicit boundaries and a verifiable end state.

<Fig caption="the anatomy of a contract" text={`
{
  "id": "http-middleware",
  "objective": "Add a rate-limit middleware to the public API router",
        ─ one sentence, ≤ 160 chars
  "outputFormat": "the detail: exact files, structure, shape of done",
        ─ written for human review
  "keyRisks": ["429 handling changes the public error surface"],
        ─ only when genuinely risky
  "ownsPaths": ["src/http/**"],         ─┐  the boundary — collisions
  "mustNotTouch": ["src/limiter/**"],    ├  die here, not on the train
  "skills": ["my-tdd-loop"],            ─┘
  "modelTier": "fast" | "standard" | "frontier",
  "dependsOn": ["limiter-core"],
        ─ dependency waves; findings + merged code flow downstream
  "expectedState": "requests over the cap get 429; pnpm test green"
        ─ what the judge scores against
}
`} />

Alongside the contracts, the planner returns a run **brief** covering
objectives, constraints, and done-when: the normalized reading of your goal,
rendered on the bridge instead of prompt prose. Contracts are validated before the plan
reaches you; problems surface as warnings on the plan-approval decision.

## Fleet vs. one agent: single-context detection

The most important planning rule is about when **not** to use the fleet.
The planner fans out only when tasks own genuinely disjoint paths and can be
verified independently. If the work is tightly coupled (shared files, deep
sequential dependencies, one subsystem, roughly under 5 files touched), it
plans exactly **one** contract:

> A single agent with full context outperforms a fragmented fleet on coupled
> work, and fan-out pays context-fragmentation costs. This is the #1
> multi-agent failure mode.

So a "fleet run" with one arm is not a degenerate case; it's the planner
doing its job.

## Boundaries are load-bearing

`ownsPaths` and `mustNotTouch` are not documentation:

<div class="k-prompt-list">
- The [judge](../judge/) fails any checkpoint whose diff writes outside the owned paths, automatically, regardless of score.
- The [findings bus](../findings-bus/) routes discoveries to siblings by owned-path overlap.
- The pre-write conflict radar (`kraken radar`) simulates merges across live worktrees, so overlapping edits surface while they're still one conversation.
</div>

Tasks whose owned paths overlap *will* collide at merge. The planner is
told so, plans around it, and you should reject plans where the partition
looks wrong.

## keyRisks: the planner names the sharp edges

For genuinely risky work (migrations, concurrency, breaking surface) the
planner attaches `keyRisks`: what could go wrong, one item each. They render
on the plan review so a human approves with eyes open, and they're omitted
when nothing is genuinely risky, so they stay meaningful.

## The other half: the worker report

A contract is a two-sided document. When an arm finishes, it files a
structured **report**, the receipt for its work:

```json
{
  "summary": "…", "filesTouched": ["…"],
  "decisions": ["…"], "gotchas": ["…"],
  "interfaceChanges": ["…"], "blockers": ["…"],
  "confidence": 0.9
}
```

The report feeds the judge (as the worker's claim, ranked below gate
evidence), the findings bus (decisions, gotchas, and interface changes are
published to siblings), and [campaigns](../../guides/campaigns/) (interface
changes compose into downstream repos' goals).
