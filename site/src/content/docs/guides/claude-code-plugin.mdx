---
title: The Claude Code plugin
description: Turn any Claude Code session into a helm, with fleet tools over the Model Context Protocol plus skills for onboarding and delegation judgment.
---

Kraken ships as a Claude Code plugin: the checkout itself is the plugin
(`.claude-plugin/plugin.json` + `.mcp.json` + `skills/`). Load it and your
session becomes a helm: it can plan runs, approve them, execute the fleet,
and drain decisions, all against the same journal as the CLI and the bridge.
The session talks to Kraken over the Model Context Protocol (MCP), the
standard for connecting agents to tools.

## Install

```sh
claude --plugin-dir /path/to/kraken
```

The plugin wires up the MCP server automatically (via the bundled
`.mcp.json`, which runs `kraken mcp` over the plugin root). Without the
plugin, the raw MCP registration works too:

```sh
claude mcp add kraken -- node --experimental-strip-types \
  /path/to/kraken/packages/cli/src/index.ts mcp --repo $(pwd)
```

## The MCP tools

| Tool | What it does |
| --- | --- |
| `start_run` | Plan a new run: decomposes a goal into task contracts. Returns the `runId`; the plan waits for approval via `decide()`. |
| `list_decisions` | List pending decisions across all runs, the queue a human (or a supervising session) drains. |
| `decide` | Resolve a pending decision: plan approval, ejection handling, failure triage. Takes an optional annotation, which becomes part of the audit trail (and a [steering note](../../concepts/decisions/#steering-notes) on retries). |
| `execute_run` | Execute an approved run: dispatch arms in parallel worktrees, merge via the speculative train. Long-running. |
| `run_status` | Current state of a run: contracts, statuses, judge scores, costs, reports, pending decisions. |
| `conflict_radar` | Pre-write conflict radar: `git merge-tree` simulation across live `kraken/*` branches, so collisions show up before they're merge failures. |
| `register_project` | Register the repo on the bridge (sidebar group + channel) without needing a first run. Call after onboarding. |

The human can watch and decide from the bridge at the same time. The journal
is shared, so decisions made there appear in the session and vice versa.

## The skills

### `kraken-init` — onboard a repository

Invoked when you ask Claude Code to put a repo on Kraken. It detects the
stack, derives gate commands from the project's own scripts, **smoke-tests
each one once** (~2 minutes; the reconcile loop re-probes trunk before every
execution anyway), writes an honest `kraken.toml`, and registers the project
on the bridge. Its hard rules: never invent gates, never overwrite an
existing config without showing a diff, and if no build system is
detectable, write `gates = []` with a warning instead of guessing. Details
in [Onboard a repository](../../start/onboard/).

### `kraken-fleet` — delegation judgment

The more interesting skill: it teaches the session **when to delegate at
all**. Delegate when the work splits into 2+ tasks with disjoint file
ownership and a verifiable end state; do it inline when it's a single-file
edit, exploratory, or needs conversation context the planner can't get.

It also encodes the craft of composing a goal, the thing that determines
run quality:

> `start_run`'s goal must be self-contained: intent, hard constraints, what
> done looks like, and sequencing hints if tasks depend on each other. The
> planner never sees this conversation. Vague delegation is the #1
> multi-agent failure mode.

Then it drives the loop: review the plan via `run_status`, approve with
annotations, `execute_run`, drain `list_decisions` when the fleet escalates
(preferring `fix-forward` with a steering note over parking), and report
outcomes honestly from `run_status`: per-task status, judge scores, costs,
and anything parked.
