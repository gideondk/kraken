Real changes rarely respect repository boundaries: the API changes, then the
SDK, then the two apps that consume it. A **campaign** realizes one intent
as a chain of single-repo runs — without ever weakening the single-repo
invariants that make runs safe.

```sh
kraken campaign "Add delete-awareness to the sync protocol end to end" \
  --repos /work/api,/work/sdk,/work/app
kraken campaign-adv <campaignId>     # advance the chain
```

## Planning: slices over a dependency graph

A plan-mode arm **reads** the given repositories (changing nothing) and
splits the intent into per-repo **slices** — at most one per repository,
ordered by dependency, forming a directed acyclic graph (DAG): dependencies
first, no cycles. Repos that need no work are omitted.
Each slice's goal is written self-contained, for that repo's own planner. A
structurally invalid plan (unknown repo, duplicate slice, dependency cycle)
throws and journals the campaign `aborted` — a bad plan leaves a corpse, not
a zombie.

## interfaceChanges composition — the only channel

Arms never touch sibling repos. Context crosses repositories through exactly
one mechanism: when a downstream slice plans, the journal-carried payload of
each dependency's finished run is **composed into its goal** — the run
title and outcome, per-task one-liners, and every `interfaceChange` and
`decision` from the workers' structured reports.

A dependency that finished `partial` adds a warning: *verify what you depend
on actually landed before building against it.*

## Approval auto-advances

Each slice's plan still pauses for human review — campaigns don't buy
autonomy by dropping gates. But approving a pending plan on the bridge
**auto-advances the campaign**: the approved run executes, and the next
slices plan as their dependencies succeed. From the CLI, the same loop is
explicit:

```sh
kraken decide <runId> plan-<runId> approve && kraken campaign-adv <campaignId>
```

`campaign-adv` is re-entrant, like everything journal-backed: it plans what
became plannable, reconciles approved runs, reports what's awaiting review
or blocked, and completes the campaign when every slice settles — `success`
when all slices succeed, `partial` when upstream failures mean some slices
can never plan.

## Aborting

A campaign that can't produce a valid plan aborts itself at planning time.
A running campaign winds down through the same decision machinery as
everything else: park the slices you no longer want, and the campaign closes
`partial` — with the journal showing exactly which slices landed and which
were declined.
