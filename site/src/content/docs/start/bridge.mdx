The bridge is Kraken's web helm:

```sh
kraken serve
```

It hosts a mobile-first UI **over the journal**: the same append-only events
the CLI and the [Model Context Protocol (MCP) server](../../guides/claude-code-plugin/)
write. There is no separate bridge state: a decision made on the bridge
appears in your Claude Code session, and a run started from the CLI shows up
live on your phone. Updates stream over server-sent events (SSE).

## What it's for

The bridge is **decision queue first, fleet state below**. Its job is the
loop you actually repeat: read the plan, approve it, watch the arms, drain
the decisions the machine couldn't settle. Runs are grouped by project in the
sidebar; each project also gets a channel where intent forms before it
becomes a run.

Take the [visual tour](../../bridge/tour/) for a walk through every surface:
live runs, decision gates with tool grants, judge verdicts, campaigns, and
the triage inbox.

Put the bridge on your tailnet and pin it to your home screen. For
lock-screen push **without any infrastructure**, set an
[ntfy.sh](https://ntfy.sh) topic in `kraken.toml` and install the ntfy app:

```toml
notify_url = "https://ntfy.sh/<your-topic>"
```

Every `DecisionRequested` (plan approvals, ejections, judge failures) POSTs
to that URL and lands on your lock screen. Any webhook works; ntfy is just
the zero-setup option.

## The discipline

The bridge renders and routes; it never executes. Fleet work happens in
arms and merges through the train. The bridge is where a human reads,
questions, and decides. That split is deliberate: the moment the helm can
edit files it becomes a slower Claude Code, and Kraken loses its reason to
exist.
